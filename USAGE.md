# USAGE — 手把手使用手册

> 假设你是一名产品/工程开发者,要用三个 AI 模型会话协作开发一个新工程。全程你只做三类事:**填一次配置、日常发口令、做偶尔的外部决策**。

## 0. 前置条件

- 一个 git 仓库(新建即可),GitHub 远端 + `gh` CLI 已认证(审阅方合并、写码方查 PR 都靠它);
- 三个能访问该仓库工作目录的 AI 编码会话(同机同目录、或各自 clone 均可);
- 按能力梯度选好模型:设计方用你手里最强的,写码方用便宜量大的,审阅方用中档(可以多个)。

## 1. 初始化(一次性,约 10 分钟)

```bash
git clone https://github.com/StevenG3/trio-kernel
cd your-new-project && git init   # 或已有仓库
bash ../trio-kernel/bootstrap.sh .
```

然后**只编辑一个文件**:`docs/kernel.config.md`,把所有 `{{占位符}}` 填掉:

| 要填的 | 例子 |
|---|---|
| 项目名/一句话定位 | 拾程——把想去的地方变成走得顺的行程 |
| 三角色模型映射 | 设计=Claude 最强档;写码=Opus;审阅=GPT-5 Codex |
| 验证单命令 | `./Scripts/ci.sh`(没有就先写个跑测试的脚本,哪怕只有一行) |
| 红线清单 | 密钥不入库 / UI 禁硬编码色值 / …(按工程定) |

再顺手把 `CLAUDE.md` 和 `docs/PROJECT.md` 里的占位符填掉,commit + push。完成。

> **验证命令的唯一硬要求**:单命令、确定性(同代码必同结果)、末行有明确 PASS/FAIL 标记。做不到这一点,审阅流程会退化成扯皮。

## 2. 日常循环(这就是全部日常)

### 第一圈完整走一遍

1. **对设计方会话说**:`读收件箱,设计`,并附上你的需求(一句话即可,例:"做一个登录页")。
   → 设计方探索现状代码、写 spec 落 `docs/designs/`、把任务(指针+摘要)**追加到 coder 收件箱队尾**、commit push。
2. **对写码方会话说**:`读收件箱,开工`。
   → 写码方先过**阻塞门**(`gh pr list` 自查,有自己未合入的 PR 就拒绝取新任务);无阻塞则取**队首**任务、删条目、拉分支、TDD 实现、写 devlog、push、开 PR、把 PR 号写进 reviewer 收件箱。
3. **对审阅方会话说**:`读收件箱,审阅`。
   → 审阅方 fail-fast:先零构建静态扫描(diff 走读/红线/契约),**发现阻塞项直接落盘打回,不跑 CI**;无阻塞才核验证据与 runner 结论、定向抽查;六维评分,全 ≥4 则 squash 合并删分支,否则整改提示词写回 coder 收件箱。
4. **打回后**:再对写码方说 `读收件箱,开工` → 它看到整改条目,在**同一分支**整改再请审(阻塞门保证它不会去取新任务)。循环到合并。

### 并行加速(FIFO 的意义)

写码方在做任务 N 的整个期间,你可以随时让设计方继续出任务 N+1、N+2 的 spec 排进队尾——**两边完全并行**,写码方合并后说一声「开工」自动吃下一个。

**水线**:FIFO 最多囤 3 个任务——spec 里的"现状"描述会随每次合并腐化,囤太多等于囤返工。更远的待办让设计方记在路线图里(一句话任务名),轮到了再写 spec。

### 你唯一需要人肉介入的时刻

- 设计方上抛**外部决策**(视觉方向小样二选一、商业裁定、优先级重排)——回一句话即可;
- 设计方与写码方对 spec 有争议时的**裁定**会自动走:写码方提请 → designer 收件箱 → 你对设计方说「读收件箱,设计」→ 它裁定并修 spec。你只在裁定需要口味/商业判断时表态。

## 3. 特殊流程速查

| 场景 | 做法 |
|---|---|
| 误对写码方发了「开工」 | 阻塞门自动挡住(有未合入 PR 就不取新任务),无需人工补救 |
| 合并后才发现代码有问题 | 审阅方把整改写进 coder 收件箱「⚡ 抢占区」——优先于设计方 FIFO 的一切任务;在审 PR 的整改则不进收件箱,跟着 open PR 走 |
| spec 有矛盾/缺关键定义 | 写码方不擅自定,写 designer 收件箱提请;设计方裁定 + 修 spec 原文 + 落裁定文档 |
| 多个审阅方 | reviewer 收件箱条目**认领制**:开审前标注「审阅中 @模型名」并 commit,防双审 |
| 借用第四个模型看看 | 客座:只做顾问式审阅,产出入 `docs/reviews/`(带模型标识),无合并权 |
| 视觉/品味类新方向 | 设计方必须先出 1-2 版小样给你目检选向,再全量 spec(铁律,防错方向铺开) |
| 换模型/模型下线 | 设计方出一份"总路线图+交接文档"(PR 粒度任务表 + 口径通则 + 裁定索引),继任读它即可接续 |

## 4. CI 配置建议(可选但强烈推荐)

1. **脚本内 fail-fast**:便宜守卫(lint/红线 grep)放最前,最贵的(UI/端到端)放最后;
2. **`--quick` 模式**:给写码方内环用(秒级~分钟级),交付前仍须一次全量;
3. **self-hosted runner**(GitHub Actions 配额告急时的解法):
   ```bash
   # GitHub 仓库 → Settings → Actions → Runners → New self-hosted runner,按指引:
   mkdir ~/actions-runner-<repo> && cd ~/actions-runner-<repo>
   # 下载解包官方 runner 后:
   ./config.sh --url https://github.com/<you>/<repo> --token <注册token>
   ./svc.sh install && ./svc.sh start     # 常驻服务,开机自启
   ```
   workflow 用 `runs-on: [self-hosted, macos]` 跑同一个验证脚本——**零配额消耗**,且构成独立于写码方工作区的第三方执行,审阅方因此可以免于本地全量复跑;
4. 同机多会话共用模拟器/浏览器等资源时,给 CI 配**专属实例**(如 iOS 专用模拟器 `SIM_NAME` 环境变量),避免与开发测试互踩导致随机失败。

## 5. FAQ

**Q:为什么写码方不能自己合并?**
独立性是这套流程的地基:审阅方被授权打回、绝不做橡皮图章,历史上多次拦下"绿了但没证明"的交付。

**Q:设计方要写代码级细节到什么程度?**
"照抄级":精确文件路径、函数签名、边界、验收断言。写码方的自由度只在 HOW(实现细节),不在 WHAT。

**Q:文档会不会越堆越多?**
永久文档(designs/devlogs/reviews)只增不删是有意的——它们是三方的共同记忆和审计轨迹;收件箱永远只有待办,处理完即删;`PROJECT.md` 只记现在时。

**Q:两个会话同时 commit 撞了怎么办?**
各角色写不同的文件面(spec/实现/审阅),冲突极少;设计方的纪律是文档一律落主干、绝不落实现分支(必要时经临时 worktree),写码方只动自己的分支。

**Q:最小可用配置是什么?**
一个 repo + 一个测试脚本 + 三个会话。runner、快照测试、多审阅方都是渐进增强,不是启动条件。
