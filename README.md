# Trio Kernel — 三角色 AI 协作开发内核

> 高能力模型做设计、低成本模型做实现、中能力模型做审阅;git 即消息总线;FIFO 任务队列并行;CI 作合并门禁。
> 从 [Rovia/拾程](https://github.com/StevenG3/RoviaDemo) 项目实战中沉淀(20 个 PR、三轮以上裁定、多次流程演化)。

## 这是什么

一套让**多个 AI 编码代理并行协作**开发同一个工程的流程内核。人类只需要:

1. 项目启动时填一份配置(`kernel.config.md`);
2. 日常对不同模型会话说三句口令之一;
3. 做设计方偶尔上抛的外部决策(视觉选向、商业裁定)。

其余一切——spec、实现、TDD、审阅、打回、合并、文档留痕——由三个角色按规则自转。

## 核心机制(五件套)

### 1. 三角色分工(按模型能力梯度)

| 角色 | 建议模型档位 | 职责 | 禁区 |
|---|---|---|---|
| **设计方** | 最高能力(设计/品味/裁定) | 出照抄级 spec + 提示词;裁定设计提请;先小样后铺开 | 不写运行代码、不审、不合并 |
| **写码方** | 低成本(执行力,量大) | 按 spec 做 TDD 实现;写 devlog 附证据;开 PR | 不改设计决策、不自审自并 |
| **审阅方** | 中能力(可多个,独立性) | fail-fast 分层审阅;六维评分;打回/合并 | 不写业务代码、不做设计 |

**多审阅方**:reviewer 收件箱条目采用**认领制**——开始审阅前把条目标注"审阅中 @<模型名>"并 commit,防双审冲突;客座模型(非正选)只做顾问式审阅,产出入 reviews/ 但无合并权。

### 2. git 即消息总线(收件箱协议)

`docs/inbox/{designer,coder,reviewer}.md` 三个收件箱;条目 = 指针 + 摘要(全文在 `docs/{designs,devlogs,reviews}/`);处理方处理完删除条目并随工作 commit。**人类的三句口令**:

| 会话 | 口令 |
|---|---|
| 设计方 | 读收件箱,设计 |
| 写码方 | 读收件箱,开工 |
| 审阅方 | 读收件箱,审阅 |

### 3. 抢占区 + FIFO 任务队列 + 阻塞门

写码方收件箱是**双区单文件**(不做双文件,避免"看哪个"歧义),取任务按固定顺序判断:

1. **阻塞门**:存在自己未合入的实现 PR → 只能继续该 PR 的整改,「开工」口令**不解除**阻塞,合并才解除(防误发口令并行取任务)。在审 PR 的整改指令**不占收件箱**——它跟着 open PR 走,阻塞门天然保证其独占与抢占。
2. **⚡ 抢占区**:合并后才发现的整改/回归/热修写这里(写入权:审阅方;设计方仅限注明理由的插队)——非空时优先于 FIFO 一切任务。这是"阻塞门管不到"的唯一缺口:问题在合并后浮现时已无 open PR,靠抢占区补位。
3. **📋 FIFO 队列**:设计方随时向**队尾**追加,与开发完全并行;只取队首,不得跳取。
4. **水线(WIP limit)**:FIFO 最多 3 个任务(抢占区不计)。spec 是易腐品——每份 spec 的"现状"节锚定具体文件/行号,每合并一个 PR 就漂移一分;满水线时设计方不写下一份 spec,待办以任务名留在路线图 backlog(未成文的条目不会腐化),消化一个写一份。

### 4. CI 门禁(fail-fast 分层)

- 验证入口必须是**单命令、确定性**脚本(如 `./Scripts/ci.sh`),写码方交付前自跑一次全量;
- 推荐配 **self-hosted runner**(零 GitHub Actions 配额消耗)在干净 checkout 自动跑同一脚本 = 独立第三方执行;
- CI 脚本内部 fail-fast:便宜守卫(静态红线/lint)最前,最贵的(UI 测试)最后;提供 `--quick` 内环模式(不可作交付证据);
- **审阅方 fail-fast**:静态扫描(diff 走读/红线 grep/契约比对,零构建)确认任何阻塞条件 → 立即打回,**禁止再跑 CI**;前序无阻塞才核验 runner 结论 + 定向抽查;本地全量复跑仅限 runner 不可用/存疑/改动触及验证链路本身。**CI 是合并的必要门禁,但执行者是 runner,不是审阅方的时间。**

### 5. 文档协议(可追溯)

- 永久文档:`docs/{designs,devlogs,reviews}/YYYY-MM-DD-HH-MM-<topic-slug>.md`,只增不删不覆盖;
- `docs/PROJECT.md` 是唯一原地更新的"宪法"(红线/当前状态/环境陷阱);
- 裁定(设计提请的判决)落盘 designs/ 并**同步修订 spec 原文**,教训归档成通则(例:凡"同输入必同输出"场景禁 `hashValue`、禁 unseeded Random)。

## 快速开始

```bash
git clone https://github.com/StevenG3/trio-kernel
cd your-new-project
bash ../trio-kernel/bootstrap.sh .    # 拷贝 template/ 到当前工程
# 编辑 docs/kernel.config.md(唯一配置入口):项目名/模型映射/验证命令/红线
# 然后开三个模型会话,分别说三句口令即可运转
```

**→ 手把手教程见 [USAGE.md](USAGE.md)**:初始化清单、日常循环逐步演示(设计→开工→审阅→打回→合并)、特殊流程速查(裁定/多审阅方/客座/交接)、CI 与 self-hosted runner 配置、FAQ。

## 配置入口

所有需要按工程定制的东西集中在 **`docs/kernel.config.md`**(见 `template/docs/kernel.config.md`):项目一句话定位、三角色模型映射、验证单命令、红线清单、分支与合并策略。角色文档(`docs/roles/*.md`)与收件箱模板开箱即用,一般不需要改。

## 经验教训(内核为什么长这样)

- **spec 必须"可客观验收"**:没参与设计的审阅方只看 spec 就能逐条判 pass/fail,做不到就重写——这是三方不互相扯皮的根基。
- **视觉/品味决策先小样后铺开**:设计方先出 1-2 版方向小样给人类目检选向,再全量出 spec,避免写码方在错误方向上铺开。
- **验证也要自证**:每个测试/脚本要亲手让它红过("能红"),防守卫假阴性。
- **确定性通则**:快照/基线类验证要求同输入恒同输出;禁 `hashValue`(进程随机种子)、禁 unseeded Random、斜向几何用法向量表达。
- **设计方提交纪律**:多会话共用一个工作目录时,写码方可能随时切分支——设计方提交文档一律先确认当前分支,必要时经临时 worktree 落 master,绝不污染实现分支。
